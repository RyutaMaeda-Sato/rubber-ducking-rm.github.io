d:T5895,<p>HCD（人間中心設計）を効果的に実践するためには、「誰のために設計するのか」を正確に理解することが不可欠です。本記事では、1次ユーザー、2次ユーザー、間接ユーザー、広義のユーザーという4つの分類を体系的に解説し、実践での活用方法を詳しく説明します。</p>
<h2>ユーザー分類が重要な理由</h2>
<h3>設計判断の精度向上</h3>
<p>ユーザーを適切に分類することで、<strong>限られたリソースを最適に配分</strong>し、各ユーザーグループに最大の価値を提供できます。曖昧な「ユーザー」という概念では、誰にとっても中途半端な設計になるリスクがあります。</p>
<h3>ISO 9241-210との関係</h3>
<p>国際規格ISO 9241-210の「利用状況の理解と明示」プロセスでは、システムに関わるすべての関係者の理解が求められています。ユーザー分類は、この要求を体系的に満たすための重要な手法です。</p>
<h2>1次ユーザー（プライマリーユーザー）</h2>
<h3>定義と特徴</h3>
<p>1次ユーザーは<strong>システムの主要な利用者</strong>であり、最も頻繁にシステムを使用してその主目的を達成する人です。</p>
<pre class="language-markdown"><code class="language-markdown code-highlight"><span class="code-line">特徴:
</span><span class="code-line"><span class="token list punctuation">-</span> 最高頻度での利用
</span><span class="code-line"><span class="token list punctuation">-</span> システムの中核機能を使用
</span><span class="code-line"><span class="token list punctuation">-</span> 主要なタスクを実行
</span><span class="code-line"><span class="token list punctuation">-</span> 設計の最優先対象
</span></code></pre>
<h3>具体例</h3>
<p><strong>ECサイトの場合</strong>:</p>
<ul>
<li>商品を検索・購入する顧客</li>
<li>商品レビューを投稿する利用者</li>
</ul>
<p><strong>医療システムの場合</strong>:</p>
<ul>
<li>診療記録を作成する担当医師</li>
<li>治療計画を立案する医師</li>
</ul>
<p><strong>営業支援システムの場合</strong>:</p>
<ul>
<li>顧客情報を管理する営業担当者</li>
<li>商談進捗を更新する営業スタッフ</li>
</ul>
<h3>設計での重要性</h3>
<p>1次ユーザーは<strong>ユーザビリティ評価の主要対象</strong>となり、ペルソナ設計の中心に据えられます。彼らのタスク効率と満足度が、システム全体の成功を大きく左右します。</p>
<h2>2次ユーザー（セカンダリーユーザー）</h2>
<h3>定義と特徴</h3>
<p>2次ユーザーは<strong>時折システムを使用する人</strong>で、1次ユーザーをサポートする役割を担います。利用頻度は低いものの、システム運用において重要な存在です。</p>
<pre class="language-markdown"><code class="language-markdown code-highlight"><span class="code-line">特徴:
</span><span class="code-line"><span class="token list punctuation">-</span> 低〜中頻度での利用
</span><span class="code-line"><span class="token list punctuation">-</span> 特定機能のみを使用
</span><span class="code-line"><span class="token list punctuation">-</span> サポート・管理業務が中心
</span><span class="code-line"><span class="token list punctuation">-</span> 専門的な要求を持つ
</span></code></pre>
<h3>具体例</h3>
<p><strong>ECサイトの場合</strong>:</p>
<ul>
<li>カスタマーサポート担当者（問い合わせ対応）</li>
<li>在庫管理者（商品情報更新）</li>
<li>経理担当者（返金・請求処理）</li>
</ul>
<p><strong>医療システムの場合</strong>:</p>
<ul>
<li>看護師（患者記録確認）</li>
<li>薬剤師（処方薬チェック）</li>
<li>医事課スタッフ（請求データ作成）</li>
</ul>
<p><strong>営業支援システムの場合</strong>:</p>
<ul>
<li>営業マネージャー（チーム実績確認）</li>
<li>システム管理者（ユーザー権限設定）</li>
<li>経理担当者（売上データ抽出）</li>
</ul>
<h3>設計での考慮点</h3>
<p>2次ユーザーには<strong>効率性重視のインターフェース</strong>と適切な権限管理が重要です。専門業務に特化した機能要求を持つことが多く、操作の正確性と迅速性が求められます。</p>
<h2>間接ユーザー</h2>
<h3>定義と特徴</h3>
<p>間接ユーザーは<strong>システムを直接操作しない</strong>ものの、その結果や影響を受ける人々です。システムの価値や制約を理解する上で重要な視点を提供します。</p>
<h3>分類と特徴</h3>
<p><strong>受益者型</strong>:</p>
<ul>
<li>システムの恩恵を受ける人</li>
<li>例: 患者（医療システムによる診療向上）</li>
</ul>
<p><strong>影響受容者型</strong>:</p>
<ul>
<li>システムの影響を受ける人</li>
<li>例: 近隣住民（配送システムの交通影響）</li>
</ul>
<p><strong>情報利用者型</strong>:</p>
<ul>
<li>システムの出力情報を利用する人</li>
<li>例: 経営陣（売上レポートの活用）</li>
</ul>
<h3>具体例：オンライン学習システム</h3>
<pre class="language-markdown"><code class="language-markdown code-highlight"><span class="code-line">1次ユーザー: 受講生
</span><span class="code-line">2次ユーザー: 教師、システム管理者
</span><span class="code-line">間接ユーザー:
</span><span class="code-line"><span class="token list punctuation">-</span> 保護者（学習進捗の確認）
</span><span class="code-line"><span class="token list punctuation">-</span> 学校管理者（教育効果の評価）
</span><span class="code-line"><span class="token list punctuation">-</span> 教育委員会（政策判断への活用）
</span><span class="code-line"><span class="token list punctuation">-</span> 地域社会（教育水準の向上）
</span></code></pre>
<h3>設計への影響</h3>
<p>間接ユーザーの要求は主に<strong>非機能要求</strong>として現れます。プライバシー保護、データ品質、社会的責任などの観点から設計制約を与えることが多くあります。</p>
<h2>広義のユーザー</h2>
<h3>包括的な概念</h3>
<p>広義のユーザーは、<strong>システムエコシステム全体に関わるすべての人</strong>を指す最も包括的な概念です。持続可能で社会に受け入れられるシステム設計には、この視点が不可欠です。</p>
<h3>構成要素</h3>
<pre class="language-markdown"><code class="language-markdown code-highlight"><span class="code-line">広義のユーザー =
</span><span class="code-line">直接ユーザー（1次・2次）
</span><span class="code-line">＋ 間接ユーザー
</span><span class="code-line">＋ その他のステークホルダー
</span><span class="code-line">├── 意思決定者（経営陣、政策立案者）
</span><span class="code-line">├── 規制当局（監督機関、認定機関）
</span><span class="code-line">├── 投資家・出資者
</span><span class="code-line">├── 地域社会・コミュニティ
</span><span class="code-line">├── 競合他社
</span><span class="code-line">└── 将来の利用者
</span></code></pre>
<h3>設計での重要性</h3>
<p>広義のユーザーを考慮することで：</p>
<ul>
<li><strong>社会的責任</strong>を果たす設計</li>
<li><strong>持続可能性</strong>の確保</li>
<li><strong>予期しないリスク</strong>の回避</li>
<li><strong>長期的な価値創出</strong>の実現</li>
</ul>
<h2>実践的な活用フレームワーク</h2>
<h3>ステップ1: ユーザー特定マトリックス</h3>
<pre class="language-markdown"><code class="language-markdown code-highlight"><span class="code-line">【影響度×関与度マトリックス】
</span><span class="code-line">            | 高関与 | 低関与
</span><span class="code-line">影響度 高   |   A    |   B
</span><span class="code-line">影響度 低   |   C    |   D
</span><span class="code-line">
</span><span class="code-line">A: 1次ユーザー（最優先）
</span><span class="code-line">B: 重要な間接ユーザー
</span><span class="code-line">C: 2次ユーザー
</span><span class="code-line">D: その他のステークホルダー
</span></code></pre>
<h3>ステップ2: 要求事項の優先順位付け</h3>
<p><strong>1次ユーザー</strong>: 機能要求の70-80%</p>
<ul>
<li>核心的なタスクとワークフロー</li>
<li>高い使いやすさと効率性</li>
</ul>
<p><strong>2次ユーザー</strong>: 専門機能と管理要求</p>
<ul>
<li>権限管理と承認フロー</li>
<li>データ管理と分析機能</li>
</ul>
<p><strong>間接ユーザー</strong>: 非機能要求</p>
<ul>
<li>セキュリティとプライバシー</li>
<li>データ品質と可用性</li>
</ul>
<p><strong>広義のユーザー</strong>: 社会的責任要求</p>
<ul>
<li>アクセシビリティ</li>
<li>環境への配慮</li>
</ul>
<h3>ステップ3: 評価指標の設定</h3>
<pre class="language-markdown"><code class="language-markdown code-highlight"><span class="code-line">1次ユーザー評価指標:
</span><span class="code-line">□ タスク完了率: 90%以上
</span><span class="code-line">□ 平均タスク時間: 前版比20%短縮
</span><span class="code-line">□ SUSスコア: 70以上
</span><span class="code-line">□ ユーザー満足度: 4.0/5.0以上
</span><span class="code-line">
</span><span class="code-line">2次ユーザー評価指標:
</span><span class="code-line">□ 専門業務の効率性
</span><span class="code-line">□ データ処理の正確性
</span><span class="code-line">□ 管理機能の使いやすさ
</span><span class="code-line">
</span><span class="code-line">間接ユーザー評価指標:
</span><span class="code-line">□ データ品質の維持
</span><span class="code-line">□ プライバシー保護の徹底
</span><span class="code-line">□ システム可用性: 99.9%以上
</span></code></pre>
<h2>実践事例：病院情報システム</h2>
<h3>ユーザー分類の全体像</h3>
<p><strong>1次ユーザー</strong>:</p>
<ul>
<li>医師: 電子カルテ作成、診断支援機能利用</li>
<li>看護師: 看護記録入力、薬剤管理</li>
</ul>
<p><strong>2次ユーザー</strong>:</p>
<ul>

<li>システム管理者: ユーザー管理、セキュリティ設定</li>
</ul>
<p><strong>間接ユーザー</strong>:</p>
<ul>
<li>患者: 診療情報の対象者、医療の質向上の受益者</li>
<li>患者家族: 病状説明の受手、面会時の情報確認</li>

<li>保険者: 診療報酬審査データの利用</li>
</ul>
<p><strong>広義のユーザー</strong>:</p>

<li>地域医療機関: 医療連携ネットワークの構築</li>
<li>社会全体: 医療の質と安全性の向上</li>
</ul>
<h3>設計への反映例</h3>
<pre class="language-markdown"><code class="language-markdown code-highlight"><span class="code-line">1次ユーザー重視の設計:
</span><span class="code-line"><span class="token list punctuation">-</span> 直感的な操作フロー

</span><span class="code-line">2次ユーザー向けの設計:

</span><span class="code-line"><span class="token list punctuation">-</span> 専門的な分析ツール
</span><span class="code-line"><span class="token list punctuation">-</span> 効率的なバッチ処理
</span><span class="code-line">
</span><span class="code-line">間接ユーザー配慮の設計:

</span><span class="code-line"><span class="token list punctuation">-</span> 長期的なデータ保存
</span></code></pre>
<h2>よくある課題と解決策</h2>
<h3>課題1: ユーザー要求の競合</h3>
<p><strong>問題</strong>: 異なるユーザー分類間での要求の矛盾</p>
<p><strong>解決策</strong>:</p>
<ul>
<li>影響度による優先順位設定</li>
<li>段階的リリースでの対応</li>
<li>設定可能なインターフェース提供</li>
</ul>
<h3>課題2: 間接ユーザーの見落とし</h3>
<p><strong>問題</strong>: 直接ユーザーのみに焦点を当てた設計</p>
<p><strong>解決策</strong>:</p>
<ul>
<li>ステークホルダーマップの作成</li>
<li>影響分析の実施</li>

<h2>まとめ</h2>
<p>HCDにおけるユーザー分類は、効果的な設計判断の基盤となる重要な概念です。1次ユーザーを中心としながらも、2次ユーザー、間接ユーザー、さらには広義のユーザーまで考慮することで、真に価値のあるシステムを設計できます。</p>
<p><strong>成功のポイント</strong>:</p>
<ul>
<li>各ユーザー分類の特徴と役割を正確に理解する</li>
<li>影響度と関与度に基づく優先順位付けを行う</li>
<li>継続的なユーザー関係の見直しと調整を実施する</li>
<li>システムエコシステム全体の持続可能性を考慮する</li>
</ul>
<p>この体系的なアプローチにより、すべてのステークホルダーにとって価値のあるシステムの実現が可能になります。</p>

<h2>問題集 解答</h2>
<p><strong>問題1：B</strong> - 1次ユーザーは「システムを最も頻繁に使用し、主目的を達成する人」として定義されます。</p>
<p><strong>問題2：C</strong> - システムの主目的を達成するのは1次ユーザーの特徴であり、2次ユーザーは主にサポート機能を担います。</p>
<p><strong>問題3：A</strong> - 間接ユーザーは「システムを直接操作しないが、その結果や影響を受ける人」として定義されます。</p>
<p><strong>問題4：B</strong> - カスタマーサポート担当者は、顧客（1次ユーザー）をサポートする役割を担う2次ユーザーです。</p>
<p><strong>問題5：C</strong> - 患者は医療システムを直接操作せず、診療の質向上という結果を受ける間接ユーザーです。</p>
<p><strong>問題6：D</strong> - システム開発者は広義のユーザーには含まれません。ユーザーは利用側の観点で定義されます。</p>
<p><strong>問題7：C</strong> - 1次ユーザーにとって最も重要なのは、システムの主目的である核心的タスクの効率性です。</p>
<p><strong>問題8：B</strong> - 間接ユーザーの要求は主に非機能要求（セキュリティ、プライバシー、データ品質等）として現れます。</p>
<p><strong>問題9：A</strong> - 影響度が高く関与度も高いのは、システムの主要利用者である1次ユーザーです。</p>
<p><strong>問題10：C</strong> - ユーザー分類の主要な役割は、システムに関わるすべての関係者を体系的に理解することです。</p>
<p><strong>問題11：D</strong> - HCDの基本原則により、すべてのユーザーは設計プロセス全体を通じて継続的に関与すべきです。</p>
<p><strong>問題12：C</strong> - 広義のユーザーを考慮する最大の利点は、システムの社会的持続可能性の向上です。</p>
